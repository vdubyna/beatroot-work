# Домашня робота: брязкіт контактів кнопки

Прошивка для ESP32-S3 `YD-ESP32-23`, яка рахує сирі interrupt-події на
`GPIO15` без програмного debounce.

Поточна версія використовує штучний брязкіт:

- реальна кнопка на `GPIO17` запускає сценарій;
- `GPIO16` через транзистор 2N2222 створює швидкі імпульси;
- `GPIO15` вимірює ці імпульси як сигнал кнопки.

## Підключення

| Пристрій | Пін | Куди |
| --- | --- | --- |
| Реальна кнопка | одна ніжка | `GPIO17` |
| Реальна кнопка | друга ніжка | `GND` |
| Резистор pull-up | один кінець | `3V3` |
| Резистор pull-up | другий кінець | `GPIO15` |
| 2N2222 | collector | `GPIO15` |
| 2N2222 | emitter | `GND` |
| 2N2222 | base | через `4.7 кОм` до `GPIO16` |

Схема:

```text
Реальна кнопка-тригер:

GPIO17 ---- кнопка ---- GND


Штучний брязкіт:

3V3 ---- 10 кОм ----+---- GPIO15
                    |
                 collector
                  2N2222
                 emitter
                    |
                   GND

GPIO16 ---- 4.7 кОм ---- base 2N2222
```

Осцилограф для перегляду штучних імпульсів:

```text
CH1 probe tip -> GPIO15
CH1 GND       -> GND ESP32
```

`GPIO15` у спокої має бути `HIGH`. Коли транзистор відкривається, він тягне
`GPIO15` у `LOW`.

## Штучний брязкіт

Один повний цикл `натиснув -> відпустив` генерує:

- `trigger=press`: 5 фронтів на `GPIO15`;
- `trigger=release`: 5 фронтів на `GPIO15`;
- разом: 10 фронтів, з них 5 `falling` і 5 `rising`.

Тому без debounce один фізичний цикл кнопки-тригера може давати приблизно:

```text
+5 irq_falling
+5 irq_rising
+10 irq_edges
```

## Що друкує Serial Monitor

Програма рахує всі зміни стану `GPIO15`:

- `irq_edges` - усі фронти, які зловив hardware interrupt;
- `irq_falling` - interrupt-переходи `HIGH -> LOW`, аналогічно Arduino `FALLING`;
- `irq_rising` - interrupt-переходи `LOW -> HIGH`;
- `poll_edges` - зміни стану, які побачило звичайне читання GPIO раз на 1 мс;
- `poll_falling` - polling-переходи `HIGH -> LOW`;
- `poll_rising` - polling-переходи `LOW -> HIGH`;
- `last_irq_us` - час останнього interrupt-фронту з моменту старту прошивки;
- `level` - поточний стан входу.

Для порівняння з прикладом із домашки використовуй:

- `falling`, якщо натискання дає перехід `HIGH -> LOW`;
- `rising`, якщо натискання дає перехід `LOW -> HIGH`.

У цій схемі вимірюваний сигнал `GPIO15` працює як active-low, тому основний
лічильник для порівняння - `irq_falling`.

## Поточний результат

Під час перевірки штучного брязкоту мікроконтролер побачив:

```text
irq_edges=92
irq_falling=45
irq_rising=47
poll_edges=18
poll_falling=9
poll_rising=9
```

Висновок: hardware interrupt бачить швидкі імпульси брязкоту, а polling раз
на 1 мс бачить переважно тільки один перехід натискання і один перехід
відпускання.

## Запуск

З цієї теки:

```sh
cargo build
```

Прошити плату і відкрити Serial Monitor:

```sh
cargo run --bin button-bounce-counter
```

## Debounce-варіант

Другий варіант прошивки рахує ті самі raw-фронти на `GPIO15`, але додатково
має програмний debounce `5 ms`:

```sh
cargo run --bin button-bounce-debounced
```

У цьому режимі:

- `raw_falling` / `raw_rising` показують брязкіт без фільтрації;
- `debounced_presses` має збільшуватися на 1 за одне натискання кнопки-тригера;
- `debounced_releases` має збільшуватися на 1 за одне відпускання.

Перевірений результат для 3 повних циклів:

```text
raw_edges=30
raw_falling=15
raw_rising=15
debounced_presses=3
debounced_releases=3
```

Тобто кожен фізичний цикл генерує 10 raw-фронтів, але програмний debounce
залишає тільки одне натискання і одне відпускання.

Додатковий діагностичний сканер GPIO:

```sh
cargo run --bin pin-scanner
```
